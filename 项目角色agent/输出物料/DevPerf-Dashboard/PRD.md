# DevPerf Dashboard 产品需求文档（PRD）

> 版本：v1.0
> 日期：2026-04-09
> 基于调研报告 research-market.md 和用户需求规格 devperf-dashboard-spec.md 编写

## 1. 产品概述

- **产品名称**：DevPerf Dashboard（研发人效 Dashboard）
- **一句话定位**：基于 Plane + Gitea 已有基础设施的轻量级研发效能数据聚合展示层，为杰森集团管理层提供只读的团队产出透明化窗口
- **目标用户**：杰森集团 CTO、工程总监、项目组长、杰森集团管理层（投资方）
- **核心价值**：将分散在 Plane（任务管理）和 Gitea（代码托管）中的研发数据自动聚合为可视化仪表盘，替代飞书表格的手动管理模式，实现研发进度的实时透明化

### 核心原则

1. **不自建项目管理系统**：Plane 社区版负责任务管理/Sprint/看板，本项目只做 Dashboard 展示层
2. **只读优先**：Dashboard 是观察窗口，不是操作工具（OKR 编辑是唯一的写入操作）
3. **数据不出内网**：完全自托管，Docker Compose 部署，与 Plane/Gitea 同机运行
4. **过程指标仅诊断用**：代码行数、commit 频次不作为绩效打分依据，仅帮助 CTO 和组长发现瓶颈

## 2. 用户画像

### 画像 A：CTO / 工程总监

| 维度 | 描述 |
|------|------|
| 角色 | 技术团队最高负责人，管理 8 条产品线 |
| 痛点 | 每周花 2-3 小时手动汇总飞书表格了解各产品线进度；缺少全局视角发现跨团队瓶颈；向投资方汇报时数据不直观 |
| 期望 | 打开 Dashboard 10 分钟内了解全部产品线状态；一键下钻到项目/个人层面；数据自动更新无需手动维护 |
| 使用场景 | 每周一早上扫描全局；季度评审前查看 OKR 进度；杰森汇报前导出关键图表 |

### 画像 B：项目组长

| 维度 | 描述 |
|------|------|
| 角色 | 单个产品线负责人，管理 3-8 名开发者 |
| 痛点 | 站会前需要逐个查看 Plane 任务状态和 Gitea PR 情况；难以识别哪位成员需要帮助；Sprint 交付率缺少历史趋势 |
| 期望 | 在项目明细页看到团队任务分配矩阵；通过 KPI 评分卡提前发现成员阻塞；燃尽图辅助 Sprint 计划 |
| 使用场景 | 每日站会前 5 分钟查看项目状态；Sprint 结束时复盘交付率；1:1 面谈时参考个人产出数据 |

### 画像 C：杰森集团管理层（viewer）

| 维度 | 描述 |
|------|------|
| 角色 | 投资方管理人员，非技术背景 |
| 痛点 | 依赖口头汇报了解研发进度，信息滞后且不可靠；无法自主查看项目实际状态 |
| 期望 | 用只读账号随时查看团队总览和项目进度；数据维度默认按"项目"组织；能点击下钻但不能修改任何内容 |
| 使用场景 | 每周/每月查看研发产出大盘；季度评审时按项目查看进展；需要时下钻到个人产出 |

### 画像 D：系统管理员

| 维度 | 描述 |
|------|------|
| 角色 | 负责系统配置和维护的技术人员 |
| 痛点 | 新成员入职时需要手动配置 Git 作者映射；同步异常时缺少诊断手段 |
| 期望 | 在管理后台创建/管理用户；配置 Git 作者映射；查看同步日志和手动触发同步 |
| 使用场景 | 新成员入职配置；同步异常排查；杰森方 viewer 账号创建 |

## 3. 功能需求

### 3.1 用户认证（P0）

- **描述**：JWT 认证系统，支持邮箱+密码登录，Token 有效期 7 天
- **用户故事**：作为任何角色的用户，我希望用邮箱和密码登录 Dashboard，以便安全访问对应权限的数据
- **验收标准**：
  - Given 用户已注册且密码正确
  - When 提交登录表单（email + password）
  - Then 返回 JWT Token，自动跳转到团队总览页，Token 有效期 7 天
  - Given Token 过期或无效
  - When 请求任何受保护 API
  - Then 返回 401，前端自动跳转登录页
- **优先级**：P0
- **边界条件与异常场景**：
  - 密码错误时返回"邮箱或密码错误"（不泄露邮箱是否存在）
  - 连续 5 次登录失败后锁定账号 15 分钟
  - Token 刷新不在 MVP 范围内，过期后重新登录

### 3.2 团队总览页（P0）

- **描述**：默认落地页，6 个图表面板 + 顶部筛选栏，展示全局研发状态
- **用户故事**：作为 CTO，我希望在一个页面内看到全部产品线的 Sprint 交付率、任务分布、代码活动、OKR 进度，以便每周一早上 10 分钟内掌握团队状态
- **验收标准**：
  - Given 用户已登录且有查看权限
  - When 访问首页 /
  - Then 展示 6 个图表面板，数据来自最近一次同步，加载时间 < 3 秒
  - Given 用户选择时间范围筛选（本季度/上季度/自定义）
  - When 筛选条件变更
  - Then 全部 6 个面板联动刷新，刷新时间 < 2 秒
- **优先级**：P0
- **6 个面板明细**：

**面板 1：Sprint 交付率趋势**
- 图表类型：柱状图 + 80% 目标虚线
- 数据：近 6 个 Cycle 的 plannedPoints vs completedPoints
- 交互：点击柱子弹窗展示该 Sprint 的任务列表

**面板 2：任务状态分布**
- 图表类型：环形图
- 数据：Todo / In Progress / Review / Done 各状态任务数
- 交互：点击扇区侧栏展示对应状态的任务清单

**面板 3：产品线进度总览**
- 图表类型：横向进度条组
- 数据：各 Project 当前活跃 Cycle 的完成百分比
- 交互：点击进度条跳转到 /projects/:id

**面板 4：每周代码活动**
- 图表类型：堆叠面积图（按成员堆叠）
- 数据：近 12 周每周 Commits + PRs
- 交互：Hover 显示各人贡献明细

**面板 5：OKR 完成进度**
- 图表类型：水平进度条 + 百分比数值
- 数据：各 Objective 的加权进度
- 交互：点击展开显示 KR 明细

**面板 6：PR 平均合入时间**
- 图表类型：折线图 + 48h 预警虚线
- 数据：近 12 周每周 PR 平均 mergeTimeHours
- 交互：Hover 显示该周 PR 列表

- **布局**：2 列 x 3 行 Grid，响应式在移动端变为 1 列
- **边界条件与异常场景**：
  - 无数据时展示空状态占位图（"暂无数据，请等待首次同步"）
  - 筛选无结果时展示"该时间范围内无数据"
  - 同步中时面板右上角展示"数据同步中"状态标识

### 3.3 项目明细页（P0）

- **描述**：单个项目的深度数据展示，包含燃尽图、里程碑、任务矩阵、Git 活动
- **用户故事**：作为组长，我希望从总览页点击进入项目明细，看到该项目的 Sprint 燃尽图和团队任务分配矩阵，以便站会前了解项目状态
- **验收标准**：
  - Given 用户在总览页点击某个项目进度条
  - When 跳转到 /projects/:id
  - Then 展示该项目的燃尽图、里程碑时间线、任务分配矩阵、Git 活动概览
  - Given 当前 Sprint 有燃尽图数据
  - When 页面加载
  - Then 燃尽图展示理想线（虚线）和实际线（实线），日期精确到天
- **优先级**：P0
- **子模块**：
  - 燃尽图：理想线 vs 实际线
  - 里程碑时间线：已完成(绿) / 进行中(蓝) / 即将到期(橙) / 已逾期(红)
  - 任务分配矩阵：行=成员，列=状态(Todo/InProgress/Review/Done)，单元格=任务数
  - 项目 Git 活动概览：近期 Commit 数、PR 数、周趋势小图

### 3.4 个人产出页（P1）

- **描述**：个人维度的交付率趋势、Git 贡献热力图、KPI 评分卡
- **用户故事**：作为组长，我希望在 1:1 面谈前查看成员的个人产出页，了解其 Sprint 交付率趋势和 KPI 评分，以便提供有针对性的支持
- **验收标准**：
  - Given 用户有 admin 或 manager 角色
  - When 访问 /members/:id
  - Then 展示该成员的 Sprint 交付率趋势（近 6 个 Sprint 折线图）、Git 贡献热力图（近 6 个月）、KPI 综合评分卡（雷达图）
  - Given developer 角色访问
  - When 访问 /members/:id 且 id 不是自己
  - Then 返回 403，仅能查看自己的个人产出页
- **优先级**：P1
- **KPI 评分卡包含**：Sprint 交付率、平均交付周期、Bug 密度、PR 平均合入时间、Review 参与率、活跃天数
- **边界条件**：developer 仅能看自己；viewer 可通过下钻查看但导航不直接暴露

### 3.5 OKR 看板页（P0）

- **描述**：OKR 树形展示，支持内联编辑 KR 进度，自动计算 Objective 进度
- **用户故事**：作为 CTO，我希望在 OKR 看板页按季度查看所有目标和关键结果的进度，以便季度评审时有数据依据
- **验收标准**：
  - Given 用户访问 /okr
  - When 选择期间筛选（如 2026-Q2）
  - Then 展示该季度所有 Objective 的树形列表，每个 Objective 下展开 Key Results
  - Given manager 角色用户编辑某个 KR 的 currentValue
  - When 提交修改
  - Then KR 进度条更新，所属 Objective 的 progress 自动重算（加权平均），刷新延迟 < 500ms
- **优先级**：P0
- **边界条件**：admin 和 manager 可编辑 KR 进度；developer 和 viewer 只读

### 3.6 Git 活动页（P1）

- **描述**：团队整体的 Git 贡献统计，包含热力图、PR 指标表格、仓库维度统计
- **用户故事**：作为 CTO，我希望在 Git 活动页看到团队整体的代码贡献热力图和 PR 合入效率，以便识别代码审查瓶颈
- **验收标准**：
  - Given admin 或 manager 角色用户
  - When 访问 /git
  - Then 展示团队整体贡献热力图、每人 PR 指标表格、仓库维度 Commit 趋势
  - Given developer 角色
  - When 访问 /git
  - Then 仅展示自己的 Git 活动数据
- **优先级**：P1
- **权限**：viewer 角色不可访问

### 3.7 管理后台页（P0）

- **描述**：系统管理功能，包含用户管理、Git 作者映射、同步日志、手动同步触发
- **用户故事**：作为管理员，我希望在管理后台创建杰森方的 viewer 账号，并手动配置新入职成员的 Git 作者映射，以便数据归属正确
- **验收标准**：
  - Given admin 角色用户
  - When 访问 /admin
  - Then 展示用户列表（可 CRUD）、未关联的 Git 作者列表（可手动关联）、同步日志、手动同步按钮
  - Given 点击"手动触发同步"
  - When Plane 和 Gitea 同步完成
  - Then 同步日志更新，展示处理记录数和状态
- **优先级**：P0
- **权限**：仅 admin 可见

### 3.8 数据同步引擎（P0）

- **描述**：定时从 Plane 和 Gitea 同步数据到本地 SQLite，Plane 每 15 分钟，Gitea 每 30 分钟
- **用户故事**：作为 CTO，我希望 Dashboard 数据每 15 分钟自动刷新，以便看到接近实时的研发状态
- **验收标准**：
  - Given 系统正常运行
  - When 到达 Plane 同步间隔（15 分钟）
  - Then 自动拉取 Plane API 数据（Projects、Cycles、Issues、Modules），写入本地 SQLite，同步日志记录成功
  - Given Gitea 同步间隔到达（30 分钟）
  - When 执行 Gitea 同步
  - Then 拉取组织下所有仓库的 Commits 和 PRs，执行作者关联逻辑，写入本地 SQLite
  - Given 同步过程中 API 调用失败
  - When 记录错误日志
  - Then 不影响已有数据，下次同步重试，同步日志记录 error 状态和错误信息
- **优先级**：P0
- **作者关联逻辑**：按 git_email 匹配 > 按 git_username 匹配 > 按 users 表 email 精确匹配 > 未关联（等待手动处理）

### 3.9 角色权限控制（P0）

- **描述**：四级角色权限体系，admin/manager/developer/viewer
- **用户故事**：作为 admin，我希望杰森方 viewer 账号只能查看团队总览和项目明细，不能修改任何数据或访问管理后台
- **验收标准**：
  - Given viewer 角色用户登录
  - When 尝试访问管理后台或 Git 活动页
  - Then 返回 403 或导航中不展示对应入口
  - Given viewer 角色在团队总览页点击成员名字
  - When 下钻到个人产出页
  - Then 可查看（只读），但导航中不主动暴露"个人产出"入口
- **优先级**：P0
- **权限矩阵**：详见下方信息架构中的权限表

## 4. 非功能需求

### 性能要求
- 页面首次加载时间（FCP） < 2 秒
- API 响应时间 < 500ms（P95）
- 团队总览页 6 个面板并行加载，总渲染时间 < 3 秒
- SQLite 查询在 10 万条任务快照规模下响应 < 200ms

### 安全要求
- JWT Token 使用 HS256 签名，Secret 通过环境变量注入
- 密码使用 bcrypt 哈希（cost factor 12）
- API 路由全部经过 auth 中间件验证
- 角色权限在中间件层强制执行，不依赖前端隐藏

### 兼容性要求
- 浏览器：Chrome 90+、Firefox 90+、Edge 90+、Safari 15+
- 分辨率：最低 1280x720（桌面），375px 宽度（移动端基础可用）
- Docker：Docker Compose v2.x，与 Plane/Gitea 同 Docker Network

### 可访问性要求
- 颜色对比度 WCAG AA 标准（文字 4.5:1，UI 组件 3:1）
- 红绿色语义辅以图标/文字标识（色盲友好）
- 键盘可导航

## 5. 信息架构

### 页面结构

```
Login (/login)
  |
  ├── 团队总览 (/)               -- 默认落地页，全角色可见
  │     └── [筛选栏] 时间/项目
  │
  ├── 项目明细 (/projects/:id)   -- 全角色可见
  │     └── 从总览"产品线进度"下钻
  │
  ├── 个人产出 (/members/:id)    -- admin/manager 全部可见，developer 仅自己
  │     └── 从项目明细"任务矩阵"成员名下钻
  │
  ├── OKR 看板 (/okr)            -- 全角色可见，admin/manager 可编辑
  │
  ├── Git 活动 (/git)            -- admin/manager 可见，developer 仅自己
  │
  └── 管理后台 (/admin)          -- 仅 admin
        ├── 用户管理
        ├── 作者映射
        └── 同步日志
```

### 导航逻辑
- 左侧固定侧边栏：显示角色可见的导航项
- viewer 角色：仅显示"总览"、"OKR"两个导航项，"项目明细"和"个人产出"通过页内下钻访问
- 面包屑导航：项目明细页和个人产出页显示返回路径

### 核心用户动线
1. **CTO 周报动线**：登录 > 总览（扫描 6 面板）> 点击交付率低的项目 > 项目明细（查燃尽图+任务矩阵）> 点击阻塞成员 > 个人产出（查 KPI 卡）
2. **组长站会动线**：登录 > 直达项目明细 > 查任务矩阵 > 回总览看 OKR 进度
3. **杰森 viewer 动线**：登录 > 总览（按项目维度浏览）> 点击项目下钻 > 查看里程碑

### 权限矩阵

| 操作 | admin | manager | developer | viewer |
|------|-------|---------|-----------|--------|
| 查看团队总览 | Yes | Yes | Yes | Yes |
| 查看项目明细 | Yes | Yes | Yes | Yes |
| 查看个人产出 | 全部 | 下属 | 仅自己 | 下钻可查 |
| 查看 OKR | Yes | Yes | Yes | Yes |
| 编辑 OKR 进度 | Yes | Yes | No | No |
| 查看 Git 活动 | Yes | Yes | 仅自己 | No |
| 管理后台 | Yes | No | No | No |
| 触发同步 | Yes | No | No | No |

## 6. 设计指引

### 风格方向
- **调性关键词**：专业可信、清晰高效、数据驱动、管理层友好
- **审美方向**：轻量级扁平 + 柔和多层阴影，介于 Swarmia 的清爽和 LinearB 的专业之间
- **不是**：技术极客风（Grafana 风格）、监控大屏风（深蓝+闪烁数字）、C 端活泼风

### 配色方向
- 基于调研报告方案 A"信赖靛蓝"方向：靛蓝主色 + 琥珀强调色 + 微暖白背景
- 主色域：靛蓝系（#3B5998 附近），传达专业和信赖
- 强调色：琥珀/金黄（用于 CTA 和关键数据高亮）
- 语义色：绿(成功)、黄/橙(警告)、红(错误)
- 中性色：带蓝调的灰色系，不用纯灰
- ECharts 图表配色需要自定义（不使用默认主题）

### 布局方向
- 左侧固定侧边栏（可折叠）+ 右侧内容区
- 总览页 2x3 Grid 面板布局
- 卡片化信息组织，每个面板一个核心指标+一个图表
- 间距使用 4px 倍数系统

### 字体方向
- 英文：Plus Jakarta Sans 或同类有辨识度的无衬线字体
- 中文：HarmonyOS Sans SC 或思源黑体
- KPI 数字：开启 tabular figures（等宽数字）
- 代码/SHA：JetBrains Mono

### 动效基调
- 克制为主，Dashboard 不需要花哨动画
- 图表入场：ECharts 默认渐入即可
- 页面切换：淡入（200-300ms）
- 卡片 hover：translateY(-2px) + 阴影加深（200ms）
- 所有过渡使用 ease-out-quart，不使用 bounce/elastic

### 圆角
- 按钮：8px
- 卡片/面板：12px
- 弹窗：16px
- 头像/徽标：full (9999px)

### 参考产品
- Swarmia：留白和信息密度的平衡
- Plane：同一基础设施家族的设计语言
- LinearB：指标卡片的信息密度

### 品牌调性关键词
专业(Professional)、可信赖(Trustworthy)、高效(Efficient)、清晰(Clear)、数据驱动(Data-driven)

## 7. 范围边界

### MVP 包含
1. 用户认证（JWT 登录，4 级角色权限）
2. 团队总览页（6 面板 + 筛选栏）
3. 项目明细页（燃尽图 + 里程碑 + 任务矩阵 + Git 概览）
4. OKR 看板页（树形展示 + 手动编辑 KR 进度 + 自动重算）
5. 管理后台（用户 CRUD + 作者映射 + 同步日志 + 手动同步）
6. 数据同步引擎（Plane 15min + Gitea 30min）
7. 个人产出页（交付率趋势 + 热力图 + KPI 卡）
8. Git 活动页（团队热力图 + PR 指标 + 仓库统计）

### MVP 不包含（后续迭代）
1. KR 关联 Plane Cycle/Module 自动同步进度（当前仅手动编辑）
2. 暗色模式
3. 数据导出（PDF/Excel）
4. 钉钉/飞书消息通知集成
5. 自定义图表面板
6. 多 Workspace 支持（当前仅 jasonqiyuan）
7. AI 辅助的异常检测和趋势预测

### 明确排除
1. 任何任务管理功能（创建/编辑/拖拽任务——这是 Plane 的职责）
2. 代码审查功能（这是 Gitea 的职责）
3. 聊天/即时通讯
4. 移动端 App（Web 响应式即可）
5. 多租户 SaaS 架构

## 8. 开放问题

1. Plane 社区版 API 的请求频率限制未在官方文档中明确，同步频率可能需要根据实际测试调整
2. 杰森集团实际团队规模和产品线数量尚需确认（影响数据量级和性能预期）
3. viewer 角色的"下属"关系如何定义——是否需要在用户管理中增加"上级"字段
4. KPI 权重分配（结果层 70% / 过程层 30%）是否需要可配置化
5. 同步数据的历史保留策略——SQLite 数据量增长后是否需要归档机制
