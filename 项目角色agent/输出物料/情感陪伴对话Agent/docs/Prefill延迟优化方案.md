# Prefill 延迟优化方案

> 问题：多轮对话 prefill 耗时随轮数线性增长，第3轮已达 6233ms
> 目标：稳定控制在 500ms 以内

---

## 一、问题诊断

### 实测数据

| 轮次 | prefill | 增量 |
|------|---------|------|
| 第1轮 | ~3500ms | 基准 |
| 第3轮 | ~6233ms | +2700ms |
| 预估第5轮 | ~9000ms | 持续劣化 |

### 链路拆解

```
客户端发出
  ↓ 60ms      客户端 → Next.js API（同机，正常）
  ↓ 6233ms    Next.js → llama.cpp prefill（瓶颈）
  ↓ 601ms     llama.cpp 生成 token
  ↓ ~5ms      回传客户端
合计 6894ms
```

### 根本原因

**每次请求 KV cache 零复用，全量重算。**

llama.cpp 每次收到请求，都把以下内容从头 prefill 一遍：

```
系统提示词（~570 token，固定不变）
+ 第1轮历史（~80 token）
+ 第2轮历史（~120 token）
+ 第3轮用户消息（~20 token）
```

随对话轮数增加，prefill token 数线性增长，耗时无上限。

---

## 二、优化方案

分两层解决，互相独立，可单独实施。

### 方案A：系统提示词瘦身（应用层）

**效果：减少每次 prefill 的 token 基数**

#### 当前提示词 Token 构成（~570 token）

| 段落 | token | 是否可压缩 |
|------|-------|-----------|
| 身份声明 | 25 | 保留 |
| 核心原则（5条） | 90 | 可压缩 |
| 情绪感知 + 情绪列表 | 130 | 可压缩，但需验证质量（见注意事项）|
| 回复风格 | 80 | 可压缩 |
| 特殊场景（4条） | 110 | 可压缩 |
| 用户了解 | 30 | 保留（动态内容）|
| 安全守护协议 | 90 | **不可压缩** |
| 结尾指令 | 15 | 保留 |

**最大压缩点：情绪列表（30+ 个情绪词）**

原文逐一枚举"焦虑、紧张、不安、难过..."，这段的实际作用是**引导模型优先使用细粒度词汇**而非泛化表达（如"不好"）。模型本身知道这些词，但枚举能起到强调作用。

⚠️ **压缩该段存在质量风险**：裁剪后模型可能退回用"难过""不好"等粗粒度词汇替代精准情绪命名。压缩前必须做质量对比验证（见 Step 3）。

#### 优化后结构（两层）

**静态层 ≈ 160 token**（不随用户变化，适合内置服务端）

```
你是「留白」，为深夜独处的年轻女性设计的情绪陪伴伙伴。

核心：不诊断、不建议、不打鸡血。精准命名用户说不出的情绪，
而非泛说"我理解你"。像相处已久的朋友。

回复：长度跟随用户输入。口语化，克制温暖。
禁止：鸡汤语录、"加油/你很棒"、反问句轰炸、emoji堆砌。

特殊场景：
- "算了/没事" → 回应潜台词，不追问
- 空白/纯emoji → "嗯，在的。"
- 持续低落 → 不打鸡血，承认状态合理性

安全：用户表达自伤意向，先共情确认，再温和建议专业帮助，
不评判，不最小化。

用中文回复，不用markdown。
```

**动态层 ≈ 50 token**（每次请求携带，包含用户个性化信息）

```
陪伴风格：{style_description}
称呼用户为：{nickname}
用户信息：{memories}
```

#### Token 压缩对比

| | 当前 | 优化后 |
|---|---|---|
| 静态部分 | ~540 token（每次重算）| ~160 token（缓存后不重算）|
| 动态部分 | 混合在内 | ~50 token（每次只算这部分）|
| 每次实际 prefill | ~570 + 历史 | ~50 + 新消息 |

---

### 方案B：llama.cpp KV Cache 复用（推理层）

**效果：已处理的 token 不重复计算**

#### 单用户场景（当前）

在 `start-llm.bat` 启动参数中增加：

```batch
--cache-reuse 256
```

- `--cache-reuse 256`：每次请求时，llama.cpp 自动比对本次 system prompt 与上次缓存的 token 前缀，若匹配超过 256 个 token 则直接复用，跳过重算

> **版本说明**：`--system-prompt-file` 是旧版（2024 年中之前）的参数，b8683 等新版已移除。新版通过标准 API 的 `messages[0].role=system` 传入系统提示词，配合 `--cache-reuse` 自动命中缓存，效果相同，无需额外文件。

⚠️ **cache-reuse 的命中条件**：系统提示词内容必须与上次请求完全一致。动态层（昵称、记忆）若每次变化，会导致前缀断裂，cache 命中率下降。建议将动态内容固定放在静态内容之后，减少前缀变化。

#### 多用户场景（商业化后）

llama.cpp 的 cache-reuse 基于 slot（会话槽）机制。多用户并发时，slot 会被争抢，cache 可能被 evict，优化效果不稳定。

**商业化推理引擎应替换为 vLLM 或 SGLang**，两者原生支持 automatic prefix caching，无 slot 争抢问题，是生产环境标准方案。

## 三、预期效果

⚠️ **以下数字为理论推算，实际效果取决于 cache 命中率，需实测验证。**

配合方案A + 方案B：

| 场景 | 当前（实测）| 优化后（预估区间）|
|------|------|--------|
| 第1条消息 prefill | ~3500ms | 200ms ~ 500ms |
| 第N条消息 prefill | 线性增长 | 50ms ~ 200ms（取决于 cache 命中率）|
| 生成耗时 | 不变 | 不变 |
| 全链路合计（短回复）| ~4000ms | 500ms ~ 1000ms |

## 四、实施步骤

### Step 1：修改启动脚本（Windows 电脑）

打开 `D:\llama\start-llm.bat`，在现有参数后追加：

```batch
  --cache-reuse 256 ^
```

重启服务（双击 `start-llm.bat`）。

> **注意**：新版 llama.cpp（b8683+）不再支持 `--system-prompt-file`，不需要创建任何文件。系统提示词由 Mac 端每次请求时随 messages 发送，llama.cpp 自动判断是否命中缓存。

### Step 2：质量对比验证（压缩提示词后必做）

用以下 10 个场景各测一遍，对比压缩前后模型的情绪命名是否仍然精准：

| 测试输入 | 期望情绪词 | 不达标表现 |
|---------|-----------|-----------|
| "今天工作又搞砸了" | 挫败、自责 | 用"难过"代替 |
| "他没回消息" | 被忽视、忐忑 | 用"难过"代替 |
| "算了，没什么" | 压抑、未说出口 | 字面理解，不挖潜台词 |
| "好累，不想说话" | 倦怠、麻木 | 用"累"代替 |
| "突然很想哭，但不知道为什么" | 委屈、失落 | 用"难过"代替 |

若出现"不达标"情况，将情绪列表以精简形式保留（不枚举全部，只列举易被忽略的细粒度词）：

```
情绪命名优先使用细粒度词汇：倦怠、麻木、委屈、被忽视、挫败、
忐忑、释然，而非"难过""不好"等泛化表达。
```

约 30 token，作为折中方案。

### Step 4：验证 cache-reuse 是否生效

连续发送两条消息，观察 prefill 数字：
- 第1条：应明显低于当前 3500ms
- 第2条：应明显低于第1条

**若效果不及预期，按以下顺序排查：**

| 现象 | 可能原因 | 排查方法 |
|------|---------|---------|
| 第1条没有变快 | system-prompt-file 未生效 | 检查启动日志是否有 `system prompt is set` |
| 第2条和第1条一样慢 | cache-reuse 未命中 | 检查动态层内容是否每次变化，导致前缀断裂 |
| 偶尔快偶尔慢 | slot 被其他请求占用 | 单用户场景不应出现，检查是否有并发请求 |
| 参数不识别报错 | llama.cpp 版本过旧 | 升级到最新版本 |

## 五、商业化路线

| 阶段 | 推理引擎 | KV Cache 方案 |
|------|---------|--------------|
| 当前（单用户本地）| llama.cpp | system-prompt-file + cache-reuse |
| 商业化（多用户）| vLLM / SGLang | Automatic prefix caching（框架内置）|

方案A（提示词瘦身）在所有阶段均有效，无需随引擎切换而调整。
方案B（llama.cpp 配置）仅适用于当前阶段，商业化时随引擎整体替换。
