# 电脑1 · 系统提示词 KV Cache 优化操作手册

> 目标：利用 llama.cpp 的 KV cache prefix 复用机制，让固定的系统提示词只在首次对话时计算一次，之后自动命中缓存，大幅降低 prefill 延迟
> 适用版本：llama.cpp b8683 及以上新版本
> 预计耗时：5 分钟

---

## 背景：新旧版本的差异

| | 旧版 llama.cpp（2024 年中之前）| 新版 llama.cpp（b8683+）|
|---|---|---|
| 系统提示词预热方式 | `--system-prompt-file` 参数 | **已移除**，不再支持 |
| 现在的做法 | — | 通过标准 API 的 `messages[0].role=system` 传入，配合 `--cache-reuse` 自动命中缓存 |

新版的原理：只要每次请求发送的 system 消息内容完全一致，llama.cpp 就自动复用上一次的 KV cache，效果与旧版 `--system-prompt-file` 完全相同，无需任何额外配置文件。

---

## 第一步：修改启动脚本

打开 `D:\llama\start-llm.bat`，替换为以下完整内容：

```batch
@echo off
chcp 65001
title Liminal LLM Server

echo ================================================
echo  留白 LLM 服务启动中
echo  模型：Gemma 26B Q3_K_M  [GPU 全量加载]
echo  上下文：16384 tokens
echo  KV Cache：自动 prefix 复用已启用
echo  端口：8080
echo ================================================

D:\llama\llama-server.exe ^
  -m "D:\models\gemma-4-26B-A4B-it-UD-Q3_K_M.gguf" ^
  --n-gpu-layers 999 ^
  --flash-attn ^
  --cache-type-k q8_0 ^
  --cache-type-v q8_0 ^
  --ctx-size 16384 ^
  --threads 12 ^
  --threads-batch 12 ^
  --mlock ^
  --defrag-thold 0.1 ^
  --cache-reuse 256 ^
  --host 0.0.0.0 ^
  --port 8080

echo.
echo 服务已停止
pause
```

⚠️ `--threads 12` 和 `--threads-batch 12` 请替换为你的 CPU 实际核心数。
查看方法：PowerShell 输入 `(Get-WmiObject Win32_Processor).NumberOfCores`

保存文件。

---

## 第二步：重启服务

1. 如果 llama.cpp 服务正在运行，关闭那个黑色窗口
2. 双击 `D:\llama\start-llm.bat` 重新启动
3. 看到以下日志说明服务正常：

```
llama server listening at http://0.0.0.0:8080
```

**不需要** 看到 `system prompt is set`——新版本不再打印这行，属于正常现象。

---

## Windows 端到此结束，无需其他操作

系统提示词由 Mac 端应用每次请求时发送，llama.cpp 收到后自动判断是否命中缓存：

```
第1条消息：
  system prompt（~580 token）→ 计算并缓存
  用户消息（~20 token）      → 计算

第2条消息：
  system prompt（~580 token）→ 内容相同，直接命中缓存 ✅
  历史对话（~100 token）     → 内容相同，直接命中缓存 ✅
  新消息（~20 token）        → 只计算这部分
```

## 排查清单

| 现象 | 原因 | 处理方法 |
|------|------|---------|
| 服务启动后立即停止 | 参数不兼容 | 检查是否误留了 `--system-prompt-file`，新版已不支持，删除该行 |
| `--cache-reuse` 报 unknown argument | 版本过旧 | 升级 llama.cpp 到 b8683 或以上 |
| 启动日志乱码 | 编码未设置 | 脚本第二行 `chcp 65001` 必须保留 |
| prefill 没有降低 | cache 未命中 | 见下方"cache 命中说明" |

---

## Cache 命中说明

`--cache-reuse 256` 的触发条件：**本次请求的前缀 token 与上次缓存匹配超过 256 个**，则复用，跳过重算。

**会命中的情况：**
- 同一用户连续对话，系统提示词内容不变

**不会命中的情况：**
- 两次请求之间系统提示词内容有任何变化（哪怕一个字）
- 用户切换了陪伴风格或昵称，导致动态层内容变化，前缀断裂

因此，**静态层内容一旦确定，不要频繁修改**，cache 命中率才能保持稳定。

---

## 验证效果（在 Mac 端操作）

重启服务后，打开留白应用，**连续发送两条消息**，观察页面底部耗时：

```
第1条：网络 Xms · prefill Xms · 生成 Xms · 合计 Xms
第2条：网络 Xms · prefill Xms · 生成 Xms · 合计 Xms
```

**达标标准：**
- 第2条 prefill 明显低于第1条（cache 开始命中）
- 第3条及之后 prefill 稳定在低位

若第2条和第1条 prefill 相近，说明 cache 未命中，对照上方排查清单处理。
