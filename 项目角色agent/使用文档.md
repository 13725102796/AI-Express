# 项目角色 Agent 系统 — 使用文档

> 一套基于 Claude Code 的多 Agent 协作系统，将产品从"一句话需求"推进到"可运行代码"，全流程自动化。

---

## 一、系统概览

### 整体架构

```
用户一句话需求
    ↓
Phase 0（需求阶段）  ← phase0-orchestrator 编排
    ↓ 输出：PRD + 风格 demo
Phase 1（设计阶段）  ← phase1-orchestrator 编排
    ↓ 输出：页面规格 + 完整 HTML 页面
Phase 2（开发阶段）  ← phase2-orchestrator 编排
    ↓ 输出：可运行的前后端代码
```

### 角色清单

| 角色 | 文件 | 职责 |
|------|------|------|
| **编排器（3 个）** | | |
| Phase 0 编排器 | `phase0-orchestrator.md` | 协调需求阶段，串联调研→PRD→设计风格 |
| Phase 1 编排器 | `phase1-orchestrator.md` | 协调设计阶段，串联页面拆解→逐页 HTML 生成 |
| Phase 2 编排器 | `phase2-orchestrator.md` | 协调开发阶段，串联架构→开发→测试 |
| **执行 Agent（7 个）** | | |
| 调研 Agent | `research-agent.md` | 全网市场调研，输出产研报告 + 设计色彩报告 |
| 产品 Agent | `product-agent.md` | 需求澄清、PRD 编写、页面规格拆解 |
| 设计 Agent | `design-agent.md` | 设计系统规范、风格参考 demo.html |
| 页面设计 Agent | `page-design-agent.md` | 逐页生成完整交互 HTML |
| 设计审查 Agent | `design-reviewer-agent.md` | Playwright 截图 + 代码双重设计质量审查 |
| 技术架构师 Agent | `tech-architect-agent.md` | 技术选型、API 设计、数据库设计 |
| 全栈开发 Agent | `fullstack-dev-agent.md` | 前后端开发，输出可运行项目 |
| 测试 Agent | `test-agent.md` | 单元测试 + API 测试 + Playwright E2E 测试 |

### 公共协议文件（4 个）

| 文件 | 作用 |
|------|------|
| `_protocol.md` | 全局交接协议（任务派发格式、完成报告格式、审查格式、状态追踪格式） |
| `_self-check-library.md` | 各角色自检清单库（提交前必须逐项验证） |
| `_phase-transition-template.md` | Phase 间过渡文档模板（确保跨阶段信息不丢失） |
| `_dev-standards.md` | 企业级开发规范（目录结构、API 格式、数据库、安全、Git） |

## 二、严格执行指令（CRITICAL）

> **本节是整个系统的铁律。所有 orchestrator 和 agent 必须逐条遵守，不可以"为了效率"或"看起来没问题"而跳过任何一条。违反任何一条即视为流程失败。**

### 2.1 流程纪律 — 不可跳步

| 编号 | 铁律 | 违反后果 |
|------|------|---------|
| R-01 | **Phase 必须按 0 → 1 → 2 顺序执行**，不可跳过任何 Phase | 后续 Phase 缺少输入，产出质量不可控 |
| R-02 | **每个 Phase 的 Step 必须按编号顺序执行**，不可并行无依赖关系外的步骤 | 上下文丢失，Agent 产出偏离预期 |
| R-03 | **Phase 过渡文档是硬性前置条件**。Phase 1 必须读取 `phase0-to-phase1.md`，Phase 2 必须读取 `phase1-to-phase2.md`。缺失或不完整 → 停止流程，不可"凑合继续" | 跨阶段决策丢失 |
| R-04 | **每个 Step 完成后必须立即更新 status 文件**，不可批量补更新 | 断点续传失效 |

### 2.2 审查纪律 — 不可缩水

| 编号 | 铁律 | 说明 |
|------|------|------|
| R-05 | **3 轮审查是强制的，不可缩减为 1-2 轮** | 即使 Round 1 看起来完美，Round 2/3 仍须执行（聚焦更细致维度） |
| R-06 | **每轮审查必须输出结构化审查报告**，写入 `phaseN-review.md` | 无报告 = 未审查 |
| R-07 | **审查中 severity=critical 的问题必须修复后才能进入下一轮** | 不可标记为"已知问题"后跳过 |
| R-08 | **双轨审查不可缩减为单轨**。Phase 1 页面设计必须同时经过 Claude 需求审查 + design-reviewer-agent 设计审查 | 单维度审查遗漏率高 |

### 2.3 交接纪律 — 不可偷懒

| 编号 | 铁律 | 说明 |
|------|------|------|
| R-09 | **任务派发必须使用 `<task-handoff>` 完整结构**（`_protocol.md` 第 1 节）。`<context-snapshot>` 禁止为空 | 空上下文 = Agent 盲猜 |
| R-10 | **Agent 完成报告必须使用 `<task-completion>` 完整结构**。`<self-check-results>` 必须逐项回应，`<escalations>` 不可为空字符串（无问题写"无"） | 缺失报告 = orchestrator 无法验证 |
| R-11 | **交接物必须通过三步验证**（文件存在性 → 结构完整性 → 内容质量）。不可"看了一眼觉得没问题"就跳过 | Grep/Glob 验证是硬性要求 |
| R-12 | **`<self-check>` 必须从 `_self-check-library.md` 按角色复制**，不可自编、省略或简化 | 自检项是质量门禁的最后防线 |

### 2.4 开发纪律 — 不可放水

| 编号 | 铁律 | 说明 |
|------|------|------|
| R-13 | **Dev-QA 循环最多 3 轮，不可第 4 轮** | 第 3 次 FAIL 必须触发 ESCALATION，不可继续盲修 |
| R-14 | **第 2 次 FAIL 起必须执行根因分析** | 对比本次与上次差异，判断是否同一问题 |
| R-15 | **前后端类型必须对齐 `shared-types.md`** | 不可自行发明接口类型，不可"先跑通再对齐" |
| R-16 | **代码必须符合 `_dev-standards.md`** | API 响应格式、数据库字段、安全规范、项目结构全部按规范执行 |
| R-17 | **禁止提交硬编码密钥、凭证、数据库连接串** | 必须使用环境变量 + `.env.example` |

### 2.5 编排器纪律 — 不可越权

| 编号 | 铁律 | 说明 |
|------|------|------|
| R-18 | **Orchestrator 不可替代 Agent 完成工作**。不可自己写 PRD、HTML、代码 | 必须通过对应 Agent 完成，orchestrator 只做编排和审查 |
| R-19 | **设计细节由设计 Agent 决策**。间距、圆角、字号等细节不上升给用户确认 | Orchestrator 将 PRD 同步更新以保持一致即可 |
| R-20 | **Escalation 必须按影响范围分级处理**。仅当前步骤→记录继续；影响后续→暂停调整；影响整 Phase→上报用户 | 不可一律忽略或一律上报 |
| R-21 | **Agent 修改优先 SendMessage 继续（保留上下文）**。只有方向完全错误或上下文污染时才重建新 Agent | 不可每次出问题都重建，浪费上下文 |

### 2.6 执行检查清单（每个 Phase 启动前对照）

```
Phase 启动前必须确认：
□ 前置 Phase 的所有交付物文件存在且通过三步验证
□ Phase 过渡文档已读取且 <decisions> 所有字段非空
□ <file-manifest> 中列出的文件实际存在
□ status 文件已创建，所有步骤标记为 ⏳ 待开始

每个 Step 完成后必须确认：
□ Agent 返回包含 <task-completion> 且 <status> 为 completed
□ <self-check-results> 无 FAIL 项（有 FAIL 则要求 Agent 先自行修复）
□ <escalations> 已检查并按影响范围处理
□ 交付物通过三步验证（Glob + Grep + Read）
□ status 文件已更新为 ✅

每轮审查完成后必须确认：
□ 审查报告已写入 phaseN-review.md
□ critical 问题已全部修复
□ 修改指令使用 <fix-request> 格式发送
□ 修改后重新验证修复效果

Phase 结束时必须确认：
□ 所有交付物完整
□ 过渡文档已生成且通过模板验证
□ context.md 已更新关键决策
□ status 文件最终步骤标记为 ✅
```

## 三、使用方式（原第二章）

### 前置条件

- 安装 Claude Code CLI
- 将 `项目角色agent/` 目录下的 `.md` 文件配置为 Claude Code 的 Agent 定义（放在 `.claude/agents/` 目录下或项目中对应的 agent 配置位置）

### 启动命令

在 Claude Code 中通过自然语言触发对应的 orchestrator：

| 你说的话 | 触发的 Agent |
|---------|-------------|
| "做一个产品的需求和设计" / "Phase 0" / "需求阶段" | phase0-orchestrator |
| "开始设计页面" / "Phase 1" / "正式设计" | phase1-orchestrator |
| "开始开发" / "Phase 2" / "进入开发阶段" | phase2-orchestrator |

### 典型使用流程

```
1. 用户: "我想做一个个人知识库产品，支持多格式文档上传和 AI 问答"

2. Claude Code 启动 phase0-orchestrator
   → 自动调用 research-agent 调研
   → 自动调用 product-agent 写 PRD
   → 自动调用 design-agent 生成风格 demo
   → 自动执行 2-3 轮审查优化
   → 输出定稿文件 + Phase 过渡文档

3. 用户: "Phase 1，开始设计页面"
   
4. Claude Code 启动 phase1-orchestrator
   → 读取 phase0-to-phase1.md 过渡文档
   → 调用 product-agent 拆解页面规格（3 轮审查）
   → 并行调用 page-design-agent 生成所有页面 HTML
   → 每页经过 3 轮双轨审查（Claude + design-reviewer-agent）
   → 输出所有可预览 HTML 页面
   → ⛔ **暂停 — 等待用户确认设计**（不自动进入 Phase 2）

5. 【用户确认门】用户预览所有 pages/*.html 并选择一条岔路：
   - 路径 A：满意现有设计 → 回复"设计确认通过"，进入第 6 步
   - 路径 B：要换其他模型重做设计 → 在外部重做后，将新设计稿替换 `pages/`
     下的 HTML（保留文件名），再回复"设计已替换，继续 Phase 2"
   - 路径 C：仅小幅调整 → 回复具体修改点，phase1-orchestrator 继续修复
   
   用户必须显式说出"开始开发 / Phase 2 / 设计确认通过"等触发语，
   phase2-orchestrator 才会启动。phase1-to-phase2.md 必须包含
   `<user-approval>` 块，且 status 为 approved。

6. 用户: "Phase 2，开始开发"（含上面任一确认语）

7. Claude Code 启动 phase2-orchestrator
   → 读取 phase1-to-phase2.md 过渡文档
   → 调用 tech-architect-agent 做技术架构（3 轮审查）
   → 按模块流水线并行开发（fullstack-dev-agent）
   → 每个模块开发完立即测试（test-agent）
   → Dev-QA 循环修复 Bug（最多 3 轮）
   → E2E 集成测试
   → 输出可运行代码
```

## 五、Phase 1 详细流程（设计阶段）

### 流程图

```
读取 phase0-to-phase1.md
  ↓
Part A: 页面拆解
  Step 1: 前置检查（PRD + demo.html 存在）
  Step 2: [product-agent] → page-specs.md
  Step 3-5: 3 轮强制审查（页面完整性、状态覆盖、交互逻辑、假数据、组件一致性）
  Step 6: page-specs.md 定稿
  ↓
Part B: 页面设计（最大化并行）
  Step 7: 并行启动所有 [page-design-agent] → pages/P01-xxx.html, P02-xxx.html...
  Step 8-12: 3 轮并行双轨审查
    - 审查 A: orchestrator 自己做需求符合度检查
    - 审查 B: [design-reviewer-agent] 做设计质量审查（Playwright 截图验证）
    - 合并问题 → 并行修改 → 下一轮
  ↓
Part C: 交付报告 + phase1-to-phase2.md
```

### 并行策略

- 所有页面生成同时启动（`run_in_background: true`）
- 每轮审查中：所有页面 × (Claude审查 + 设计审查Agent) 全并行
- 已通过的页面跳过修改，只处理有问题的

### 交付物

| 文件 | 说明 |
|------|------|
| `page-specs.md` | 页面规格文档（定稿，经 3 轮审查） |
| `pages/P01-xxx.html` ... | 各页面完整 HTML（含交互逻辑 + 假数据，经 3 轮设计审查） |
| `phase1-review.md` | 审查报告（页面拆解 3 轮 + 每页设计 3 轮） |
| `phase1-status.md` | 流程状态追踪 |
| `phase1-to-phase2.md` | Phase 过渡文档 |

```

                   ↓
               PASS → 下一模块
               FAIL → 修复（第 2/3 次）→ 重新测试
                        ↓
                    PASS → 下一模块
                    FAIL → 第 3 次 → ESCALATION（升级处理）
```

## 七、核心机制说明

### 1. 全局交接协议（_protocol.md）

所有 Agent 间通信使用标准化 XML 格式：

- **任务派发**：`<task-handoff>` 结构，包含 task-id、上下文快照、输入文件、交付物、自检项
- **完成报告**：`<task-completion>` 结构，包含状态、自检结果、关键决策、上报问题、下游上下文
- **审查反馈**：`<review-issue>` 结构，包含 severity（critical/major/minor/suggestion）、位置、复现步骤、修复建议
- **修改指令**：`<fix-request>` 结构，合并多来源审查结果

### 2. 交接物验证（三步验证）

每次收到 Agent 输出后，orchestrator 执行：
1. **文件存在性**：Glob 确认文件存在且非空
2. **结构完整性**：Grep 搜索必要标记（如 PRD 必须含 `## 3` `## 5` `## 6`）
3. **内容质量**：读取关键 section 验证非空且符合预期

验证失败 → SendMessage 继续原 Agent 补充（不重建）

### 3. Phase 过渡文档

Phase 之间通过结构化过渡文档交接：
- `phase0-to-phase1.md`：包含产品决策、配色方案、字体方案、文件清单、预编译上下文
- `phase1-to-phase2.md`：包含页面清单、共享组件、设计令牌、审查结果、遗留问题

下一 Phase 的 orchestrator 在 Step 0 必须读取并验证过渡文档完整性。

### 4. 状态追踪与断点续传

- 每个 Phase 维护 `phaseN-status.md` 实时状态文件
- 每完成一个步骤立即更新状态
- 流程中断后可从 status 文件恢复，跳过已完成步骤

### 5. 智能降级策略

| Agent | 失败场景 | 降级方案 |
|-------|---------|---------|
| research-agent | 调研输出不完整 | 跳过调研，product-agent 自行轻量调研 |
| product-agent | PRD 不完整 | orchestrator 生成简化 PRD 骨架 |
| design-agent | 设计输出失败 | 重试 1 次，仍失败则输出纯文字设计规范 |
| fullstack-dev（后端）| API 实现失败 | 先实现 mock API，前端暂用 mock |
| test-agent | Playwright 环境问题 | 降级为手动测试清单 |

### 6. Agent 继续 vs 重建决策

- **优先 SendMessage 继续**：小问题修补、审查反馈修改、已有解决方案的 escalation
- **重建新 Agent**：方向完全错误、上下文被污染（连续 2+ 轮修复同一问题未果）

## 八、输出目录结构

所有物料统一输出到 `项目角色agent/输出物料/[项目名称]/`：

```
输出物料/[项目名称]/
├── context.md                 # 项目上下文（跨 Phase 累积）
```

## 九、质量保障机制总结

| 机制 | 说明 |
|------|------|
| **3 轮强制审查** | 每个关键环节（PRD、页面规格、页面设计、技术架构）都经过 3 轮审查 |
| **双轨审查** | 页面设计同时由 Claude（需求符合度）和 design-reviewer-agent（设计质量）审查 |
| **自检清单** | 每个 Agent 提交前必须通过角色专属自检清单 |
| **交接物验证** | 三步验证（存在性→结构→内容） |
| **Dev-QA 循环** | 开发完立即测试，最多 3 轮修复，超限触发 ESCALATION |
| **共享类型契约** | `shared-types.md` 是前后端唯一类型来源，防止接口不一致 |
| **企业级开发规范** | `_dev-standards.md` 统一目录结构、API 格式、安全标准 |
| **Phase 过渡文档** | 结构化交接确保跨阶段信息不丢失 |

## 十、快速参考

### 从 0 到 1 完整流程

```bash
# 1. 进入项目目录
cd your-project

# 2. 在 Claude Code 中说出需求，系统自动执行 Phase 0
"我想做一个 XXX 产品，核心功能是 ..."

# 3. Phase 0 完成后，确认 PRD 和 demo.html，启动 Phase 1
"Phase 1，开始设计页面"

# 4. Phase 1 完成后，确认页面设计，启动 Phase 2
"Phase 2，开始开发"

# 5. Phase 2 完成后，进入输出目录运行项目
cd 项目角色agent/输出物料/[项目名称]/code/
# 按 tech-architecture.md 中的启动命令运行
```

### 用户介入点

- **Phase 0 结束后**：确认 PRD 和设计风格方向
- **Phase 1 结束后**（强制门）：预览 HTML 页面，**必须显式确认**设计后才能进入 Phase 2。可选岔路：
  - ✅ 通过 → 直接进入 Phase 2
  - 🔄 换模型重做 → 用其他模型/工具重新生成设计稿，替换 `pages/*.html` 后再确认
  - ✏️ 小幅修改 → 提交修改清单，phase1-orchestrator 继续修复
- **Phase 2 中遇到 ESCALATION**：决定是简化需求还是换技术方案
- **Phase 2 结束后**：运行项目，验收最终产品

### Phase 1 → Phase 2 的强制确认门（重要）

为支持"换其他模型重新定义设计"的工作流，Phase 1 结束后系统**必须**暂停：

1. phase1-orchestrator 完成所有页面后，输出 `Phase 1 交付清单` 并暂停，
   不会自动生成 `phase1-to-phase2.md`，也不会触发 phase2-orchestrator
2. 用户在 `pages/` 目录下逐页预览，决定上文三条岔路之一
3. 用户回复确认语后，phase1-orchestrator 才生成 `phase1-to-phase2.md`，
   并在文档头部写入 `<user-approval>` 块（含确认时间、确认语、设计来源
   说明：是 page-design-agent 原版还是用户外部替换）
4. phase2-orchestrator 在 Step 0 读取过渡文档时，**必须**校验
   `<user-approval>` 存在且 status=approved，否则停止流程并提示用户确认
